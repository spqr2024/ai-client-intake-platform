The problem

A contact form collects a name, an email and a paragraph of free text. Then someone spends a week of evenings emailing back and forth to find out what the project actually is, what the budget is, and whether it is worth a call at all.

📭

Forms do not ask follow-ups

"I need a website" and "I need a 40-page storefront with Stripe by March" arrive in the same box, indistinguishable until someone reads and replies.

🌙

Prospects arrive off-hours

The gap between an enquiry and the first reply is when interest cools. An agent that answers immediately does not have that gap.

🧾

Qualification is manual

Budget, timeline, scope and urgency get re-derived by hand for every lead — inconsistently, and never in a form the pipeline can sort.

What's in the box

Every external dependency sits behind an interface with an offline fallback, so the whole product works before you have a single credential.

💬

AI chat widget

SSE streaming, typing indicator, quick replies, file uploads, EN/UK auto-detection, and white-label colors and bot name per workspace.

🔀

Visual workflow builder

Compose intake flows from step cards with branching rules and live structural validation — unreachable steps, loops and dead ends — plus 5 industry templates and a dry-run simulator.

🧠

Prompt management

Versioned prompts with one-click activate and rollback, an offline test bench, and per-workflow assignment.

🤖

Multi-provider AI

OpenAI, Anthropic, Gemini, OpenRouter — or the deterministic mock. Switchable at runtime per workspace, no redeploy.

📚

Document knowledge base

Upload PDF / DOCX / Markdown / TXT with paragraph-aware chunking, hybrid semantic + lexical retrieval, version history, and analytics on questions the KB could not answer.

🔗

CRM integrations

HubSpot, Pipedrive, Notion, Salesforce and generic-webhook adapters with queue-backed export, retries and a per-lead sync log.

📋

Kanban CRM

Drag-and-drop pipeline with custom stages, tags, priorities, follow-up reminders, internal comments, activity timeline and full-text search.

🔔

Notification center

In-app bell, email and Telegram behind one dispatch API, with exponential-backoff retries and a per-message delivery log.

🏢

Multi-tenant by construction

Isolated leads, KB, workflows, prompts, settings, branding and audit per workspace — cross-tenant reads return 404, not 403.

Architecture

A deterministic core with AI at the edges: the intake flow is a JSON state machine, and the LLM only rephrases, summarizes and compresses. That is what makes zero-key operation and unflaky tests possible.

Visitor · Manager → Next.js 15 → FastAPI → Postgres / SQLite + Redis · optional → Telegram · Email · LLM

Deterministic core

The state machine drives the interview. No API key means a fully working product, not a degraded one.

Everything behind an interface

LLMProvider, EmbeddingProvider, VectorStore, EmailProvider, CacheBackend, the task queue and notification channels — each with an offline fallback and a production implementation.

Stateless API

Conversation state, memory and queues live in the database or Redis, so replicas scale horizontally.

Quick start

DEMO_MODE=true provisions a populated demo workspace on first start — 12 leads across the pipeline, chat transcripts, analytics and a knowledge base — so the dashboard looks alive the moment you clone it.

Docker

# Postgres + Redis + backend + frontend
git clone __REPO_URL__.git
cd ai-client-intake-platform
cp .env.example .env
docker compose up --build

Local dev

# Backend — http://localhost:8000 (Swagger at /docs)
cd backend
python -m venv .venv && source .venv/bin/activate
pip install -r requirements-dev.txt
uvicorn app.main:app --reload

# Frontend — http://localhost:3000
cd frontend && npm install && npm run dev

Open localhost:3000 for the chat widget, /admin for the dashboard. Adding real credentials later? make doctor verifies every configured integration against the live provider before you deploy, and python -m app.telegram_bot info does the same for the bot.

How it was built

The design constraints that shaped the codebase — each one chosen because the obvious alternative failed in a specific, boring way.

01

Determinism first

The interview is a state machine, not a prompt. Tests never call a model, so the suite is fast and cannot flake on a provider's mood.

02

Offline fallback everywhere

Mock LLM, hashing embedder, in-memory cache and queue. A fresh clone is a working product, which makes the demo honest.

03

Tenancy as an invariant

Every query is workspace-scoped. Cross-tenant reads return 404 rather than 403, so existence is never leaked.

04

Fail closed, not open

The Telegram webhook rejects every request while its secret is unset, and the API refuses to boot in production on placeholder credentials.

05

Delivery is logged, not hoped

Every outbound message is a row with status, attempts and error — so "did the client get it?" is a query, not a guess.

06

The gate is the reviewer

CI enforces lint, format, types, a coverage floor, a dependency audit and a real migration + smoke test on every push.
